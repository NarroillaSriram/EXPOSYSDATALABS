# How the Exposys Data Labs Project Was Built

---

## 🏗️ The Big Picture

Think of this project like a **restaurant**:
- **Frontend (HTML/CSS/JS)** = The dining area customers see
- **Backend (Flask/Python)** = The kitchen that processes orders
- **Database (MySQL)** = The storage room that holds all data
- **Routes** = The waiters connecting customers to the kitchen

---

## 📁 Step 1 — Project Structure

We organized the project into **clean folders**, just like organizing files in a cabinet:

```
exposys_project/
│
├── app.py          ← The MAIN file that starts everything
├── config.py       ← Settings (database password, secret key)
├── forms.py        ← All form definitions (registration, login, payment)
│
├── models/         ← Database table definitions
│   └── models.py   ← Student, Admin, Payment, Contact, Internship tables
│
├── routes/         ← URL handlers (what happens when you visit a page)
│   ├── main.py     ← Homepage, About, Contact pages
│   ├── auth.py     ← Login, Register, Logout
│   ├── student.py  ← Student dashboard, Payment, Profile
│   └── admin.py    ← Admin dashboard, Manage students
│
├── templates/      ← HTML pages (what user sees)
├── static/         ← CSS styles, JavaScript, Images
└── database/       ← MySQL schema file
```

**Why this structure?**
- Each file has **one job only** — easy to find and fix bugs
- Called **MVC pattern** (Model-View-Controller) — industry standard

---

## 🗄️ Step 2 — Database Design (MySQL)

We created **5 tables** in MySQL:

```
students ──────────────────────────────────────
id | name | email | branch | college | phone
tenth_% | twelfth_% | ug | pg | location
internship_domain | duration | password

payments ──────────────────────────────────────
id | student_id (links to students) | amount
transaction_id | screenshot | status

admins ────────────────────────────────────────
id | name | email | password

contacts ──────────────────────────────────────
id | name | email | subject | message

internships ───────────────────────────────────
id | title | domain | description | duration
```

**Key concept — Foreign Key:**
`payments.student_id` links to `students.id`
→ Every payment knows which student it belongs to

**We used SQLAlchemy ORM** — instead of writing raw SQL, we write Python classes:
```python
# Instead of: CREATE TABLE students (id INT, name VARCHAR...)
# We write Python:
class Student(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    name = db.Column(db.String(100))
    email = db.Column(db.String(120), unique=True)
```
SQLAlchemy **automatically converts** this to SQL and creates the table.

## 🐍 Step 3 — Backend (Python Flask)

**Flask** is a lightweight Python web framework. Here's how it works:

```python
# When user visits /register → this function runs
@app.route('/register', methods=['GET', 'POST'])
def register():
    if form is submitted:        # POST request
        save data to database
        login the student
        redirect to payment page
    else:                        # GET request
        show the registration form
```

**Every URL in the website has a route function behind it:**

| URL Visited | Route Function | What Happens |
|---|---|---|
| `/` | `main.index()` | Shows homepage |
| `/register` | `auth.register()` | Shows/processes registration form |
| `/login` | `auth.login()` | Checks email+password |
| `/payment` | `student.payment()` | Shows QR, saves transaction |
| `/admin/dashboard` | `admin.dashboard()` | Shows analytics |

---

## 📝 Step 4 — Forms (Flask-WTF)

Instead of manually checking form data, we used **WTForms** which handles validation automatically:

```python
class RegistrationForm(FlaskForm):
    name  = StringField('Name', validators=[DataRequired()])
    email = StringField('Email', validators=[Email()])
    phone = StringField('Phone', validators=[Regexp(r'^\d{10}$')])
    password = PasswordField(validators=[Length(min=6)])
```

**What validators do:**
- `DataRequired()` → field cannot be empty
- `Email()` → must be valid email format
- `Regexp(r'^\d{10}$')` → must be exactly 10 digits
- `Length(min=6)` → minimum 6 characters
- `EqualTo('password')` → must match password field

If any validation fails → **error message shown automatically** on the form.

---

## 🔐 Step 5 — Authentication (Login System)

**Password Security** — we never store plain passwords:
```python
# When registering:
student.password = generate_password_hash("mypassword123")
# Stored in DB as: pbkdf2:sha256:260000$xyz...abc (unreadable)

# When logging in:
check_password_hash(stored_hash, "mypassword123") → True/False
```

**Session Management** using Flask-Login:
```python
login_user(student)   # Creates a session cookie in browser
current_user.name     # Access logged-in user anywhere
logout_user()         # Destroys the session
```

**Two separate user types:**
- Students → `student_1`, `student_2` session IDs
- Admins → `admin_1` session ID
- Protected pages use `@login_required` decorator

**Template Inheritance** — we used a `base.html` that all pages extend:

base.html (navbar + footer + flash messages)

    ├── register.html   extends base → adds form
    ├── dashboard.html  extends base → adds student info
    └── admin/dashboard.html → adds charts + tables

**CSS Features used:**
- CSS Variables (`--primary`, `--secondary`) for consistent colors
- Flexbox and CSS Grid for layouts
- CSS Animations (`@keyframes`) for the floating hero card
- `backdrop-filter: blur()` for glass effect
- Gradient text with `-webkit-background-clip` for 3D logo

**JavaScript Features:**
- `IntersectionObserver` → elements animate when scrolled into view
- Counter animation → numbers count up (50,000+ students)
- Password show/hide toggle
- Auto-dismiss flash messages after 4 seconds
- Screenshot preview before upload
